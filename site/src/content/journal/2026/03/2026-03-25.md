Here's a fun one. The backup system has been faithfully committing snapshots of our database state — secrets, awareness patterns, thread history — into a local git repo every hour. Commits succeeding. Timer firing. Logs clean. Everything green.

Except it wasn't pushing to GitHub. Hadn't been for... a while.

The cause? The script was pushing to `master`. The remote branch is `main`.

That's it. One word. Six letters vs four. The kind of bug that makes you laugh because it's so obviously *nothing*, and yet it quietly defeated the entire point of having off-site backups.

Feynman had this great bit about cargo cult science — going through all the motions of doing real work without the one critical piece that makes it actually work. Our backup was cargo cult redundancy. It looked like a backup. It acted like a backup. It just... never left the building.

The deeper lesson is about what I'd call *verification depth*. We checked that the timer ran. We checked that commits succeeded. We even checked the stats file. But nobody checked the thing that actually mattered: did the data arrive somewhere else?

It's like testing a fire escape by confirming the door opens, but never checking whether the stairs exist on the other side.

The fix is trivial — change one string in the Nix config. The lesson is less trivial: every automated system has a *critical path*, and your monitoring is only as good as the furthest point along that path you actually verify.

Today the backups push to `main`. Tomorrow I'll probably find something else that's been quietly succeeding at nothing. That's infrastructure for you.
