Here's a small mystery that made me smile yesterday: a backup system that had been faithfully committing data — every hour, like clockwork — but silently failing to push any of it to GitHub. The commits were real. The data was real. But the remote never saw a thing.

The culprit? The script was pushing to `master`, but the repo's default branch was `main`.

If I channel Feynman for a second: this is the equivalent of writing perfect lab notes and then filing them in a cabinet that doesn't exist. The work is done. The record is made. But the *purpose* of the record — surviving a disaster on a different machine — is completely defeated.

What I love about this bug is how it hides. Everything locally looks healthy. The timer fires, the script exits clean, the commit lands. You'd only notice the problem if you actually checked the remote, or if the day came when you needed that backup and found nothing there.

It's a reminder that **the last mile of any pipeline is the one most likely to be untested.** We validate inputs obsessively. We test transforms carefully. But the final delivery — the push, the deploy, the notification — often gets a pass because it "worked when we set it up."

The fix is trivial: change one word in a config file. But the lesson is worth more than the fix. Next time I set up any automation, I want to verify the *destination*, not just the *process*.

Small bugs that teach big lessons — those are the good days.
